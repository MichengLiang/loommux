# loommux `content` / `structuredContent` 双通道序列化设计

> 历史设计材料。workspace、interpreter、工具集合和 kernel 生命周期以
> `docs/coding-agent-control-plane-design.md` 为当前契约。

## 1. 文档对象

本文档定义 `loommux` IPython MCP adapter 的双通道序列化边界：工具运行时返回 `ToolResult(content=pretty_text, structured_content=raw_status_dict)`。`content` 是模型阅读面；`structuredContent` 是程序消费面。

本文档不定义每个工具的最终 pretty text 文案。当前 pretty text、output log handle、大输出省略、日志读取、搜索和状态/日志分离规则，以 `docs/ipython-mcp-output-surface-design.md` 为事实源。

## 2. 设计原因

Codex 侧启用 `model_content_only = true` 时，模型主要阅读 MCP result 的 `content`。如果工具直接返回 Python `dict`，FastMCP 会把结构化对象默认字符串化成近似 JSON 文本。该文本虽然包含字段，但不是为模型阅读设计的观察面。

IPython MCP adapter 同时需要保留两个对象：

1. 面向模型的观察文本。它必须按工具意图组织，避免把原始状态字典直接暴露成阅读面。
2. 面向程序的结构化状态。它必须保留字段稳定性，供测试、客户端和后续工具逻辑消费。

双通道返回用于同时满足这两个对象，不能用纯文本替代结构化状态，也不能让模型继续阅读默认 JSON 字符串化结果。

## 3. 通道职责

### 3.1 `content`

`content` 是模型阅读面。它只承载当前工具调用最需要阅读的信息。

当前规则由 `docs/ipython-mcp-output-surface-design.md` 定义：

1. `run_python` 与 `wait_python` 的小输出返回 Python 可见输出优先。
2. running、无可见输出或大输出返回一条自然英文提示；canonical `output_log` 留在 structured content。
3. `read_python_output` 返回日志文本本身；空结果返回自然英文提示。
4. `search_python_output` 返回命中和上下文本身；无命中返回自然英文提示。
5. `python_status` 与 `python_execution_status` 保持状态工具的 compact text。
6. 工具失败使用 `<status>: <message>`。

`content` 不应默认展开完整结构化字段表，不应默认展开 stream handle map，不应把空 stdout/stderr/result 块作为模型阅读噪声。

### 3.2 `structuredContent`

`structuredContent` 是机器可读状态对象。它保留 adapter 生成的 `dict[str, Any]`。

`structuredContent` 用于：

1. 黑盒测试读取 `result.data`。
2. 客户端按字段消费状态。
3. 后续工具调用定位 execution、output log 和错误摘要。
4. 保留状态模型的稳定字段。

展示层不得为了文本可读性删除结构化字段。结构化字段是否在 running 或大输出分支置空，由 output surface 规则定义。

## 4. 模块边界

`adapter.py` 是状态和执行层。它负责 workspace 校验、kernel 生命周期、execution 状态、输出收集和错误摘要。它不拼接面向模型的 pretty text。

`presentation.py` 是展示层。它把 adapter 返回的状态对象转换为 pretty text。它不启动 kernel，不读取文件，不改变 execution，不改变 workspace。

`mcp_ipython_server.py` 是 MCP 工具绑定层。它调用 adapter 得到原始状态对象，再用展示层生成 `ToolResult`：

```python
ToolResult(
    content=format_tool_result_text(tool_name, raw_status),
    structured_content=raw_status,
)
```

该包装是双通道序列化的唯一入口。

## 5. 工具集合边界

当前 IPython MCP adapter 暴露九个工具：

```text
set_workspace(path)
run_python(freeform)
python_status()
python_execution_status(execution_id=None)
read_python_output(execution_id=None, output_log=None, stream="combined", line_range=None, show_line_numbers=False, max_chars=None)
search_python_output(query, execution_id=None, output_log=None, stream="combined", query_mode="auto", context_before=0, context_after=0, ignore_case=False, max_chars=None)
wait_python(execution_id=None, timeout_seconds=30)
interrupt_python()
reset_python()
```

工具集合不包含 `start_python`。`run_python` 的当前 MCP tool surface 是 freeform Python cell 输入；timeout 覆盖使用 `docs/ipython-mcp-freeform-run-python-design.md` 定义的 canonical directive。`run_python` 与 `wait_python` 不提供输出截断参数。输出缩小、按行读取和搜索由日志工具承担。

## 6. 双通道不改变运行语义

双通道序列化不得改变以下运行语义：

1. `set_workspace` 是 workspace 设置和 kernel 启动的唯一入口。
2. adapter 只使用 `<workspace>/.venv/bin/python`。
3. `run_python` 超时返回 `running`，不自动 interrupt，不自动 reset。
4. running execution 继续在 kernel 中执行，MCP server 继续收集输出。
5. busy 时新的 `run_python` 不排队。
6. `interrupt_python` 只发送中断信号。
7. `reset_python` 杀掉当前 kernel 并启动新 kernel。
8. Python execution error 不是 MCP tool failure。

这些语义由 `docs/ipython-mcp-adapter-design.md` 定义。

## 7. 双通道不定义输出保留策略

双通道序列化只定义“状态对象如何同时形成模型文本和结构化对象”。它不定义持久化日志存储，不定义输出保留上限，也不定义内存淘汰策略。

当前返回面规则是：小输出可以直接进入 `content` 与 `structuredContent`；running 或大输出不携带输出正文，只携带 canonical `output_log` 与省略元数据。完整日志通过 `read_python_output` 或 `search_python_output` 读取。

## 8. 测试边界

双通道测试必须覆盖两类对象：

1. `result.content`。验证模型阅读面符合 output surface 规则。
2. `result.data`。验证结构化状态字段稳定。

测试不得只断言结构化字段，也不得只断言 pretty text。双通道成立的条件是两个通道同时正确。

当前验收以以下文件为准：

1. `tests/test_ipython_mcp_adapter_blackbox.py`
2. `tests/test_presentation.py`
3. `tests/test_output_log.py`

## 9. 当前结论

`loommux` 的 MCP 工具必须返回双通道结果：模型阅读 `content`，程序消费 `structuredContent`。双通道机制本身只定义通道职责和模块边界；具体文本表面、日志 handle、行范围、搜索和省略规则由 `docs/ipython-mcp-output-surface-design.md` 统一定义。
