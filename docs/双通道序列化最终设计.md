# loommux 的 `content` / `structuredContent` 双通道序列化最终设计

这份设计只回答一件事：在 `Codex` 侧已经打开 `model_content_only = true` 的前提下，如何把 `loommux` 这个 IPython MCP adapter 的七个工具，从“默认把 Python `dict` 自动字符串化成近似 JSON 文本”的状态，改造成“给模型看的 `content` 是经过设计的 pretty text，给程序看的 `structuredContent` 仍然保持稳定状态对象”的状态。

我先直接给结论，不绕。

最终方案采用双通道返回：每个工具在运行时都显式返回 `ToolResult(content=pretty_text, structured_content=raw_status_dict)`。其中 `pretty_text` 是给模型直接阅读的主叙事面，必须是中文主叙事、稳定顺序、快速扫读、完整保留关键信息的文本；`raw_status_dict` 保持当前 `adapter` 已经生成的状态字典，不破坏现有黑盒测试、程序消费路径和设计文档定义的字段表面。

这不是把结构化返回取消掉，也不是让工具彻底改成只回字符串，而是明确承认这个系统天然就有两条职责完全不同的输出通道：`content` 是给模型读的，`structuredContent` 是给程序和测试消费的。既然 `model_content_only = true` 打开后模型不会优先看 `structuredContent`，那 `content` 就不能再是 `dict` 的副产品，而必须升格为一等输出面。你前面反复强调的正是这件事，我现在把它收束成可执行设计。

---

## 1. 设计目标

这次改动的目标不是改变 `loommux` 的执行语义，不是改变 `adapter` 的状态机，不是改变七个工具的输入输出字段集合，也不是做一层花哨包装。目标非常具体：

1. 让 `Codex` 在 `model_content_only = true` 模式下，看到的是可读性强、信息结构稳定、对 agent 友好的文本。
2. 保留当前所有结构化状态字段，避免破坏测试、调用端逻辑、文档契约和后续自动化能力。
3. 不引入新的外部工具参数，不引入新的协议分支，不改变已有七个工具集合。
4. 不改变 `adapter` 当前负责的对象边界。`adapter` 继续只负责状态、执行和错误，不负责面向模型的展示文案。
5. 把“工具做什么”和“工具怎么向模型说它做了什么”两件事解耦。

这个目标组里最重要的一条是第二条。因为当前项目已经有一个明确的状态契约：设计文档把每个工具的返回形状写出来了，黑盒测试也是按 `result.data` 去读结构化字段。如果为了做 pretty text 而直接把工具返回改成纯字符串，那文本面是好看了，但程序契约会被拆掉，等于用一个问题去解决另一个问题。这不是我这次要做的。

---

## 2. 约束条件

这次设计必须服从四类约束。

第一类约束来自 `Codex` 侧配置。`~/.codex/config.toml` 里 `loommux_ipython` server 已经打开了 `model_content_only = true`。这意味着模型在工具结果消费时，主观察面就是 `content`。因此如果 `content` 仍由 FastMCP 默认把 `dict` 序列化成 JSON 样子，模型实际读到的仍然是生硬、冗余、缺乏层级的文本。换句话说，这个配置不是附属信息，它直接决定了“是否必须专门设计 `content`”。答案是必须。

第二类约束来自 `FastMCP` 的结果模型。`FastMCP` 的工具结果天然支持双通道：`content` 是传统内容块，`structured_content` 是机器可读 JSON 对象。官方代码和文档已经非常明确地给出建议：如果要完全控制 serialization，应直接返回 `ToolResult`。我已经用最小探针验证过这件事：工具即使保留 `-> dict[str, Any]` 这种返回注解，运行时仍然可以实际返回 `ToolResult(content='hello', structured_content={...})`；client 侧 `result.data` 仍然能拿到结构化字典，而 `result.content` 会单独拿到文本块。这意味着我们完全有能力把漂亮文本和结构化数据同时保留。

第三类约束来自 `loommux` 当前文档与测试。`ipython-mcp-adapter-design.md` 已经规定了七个工具的输入和状态字段，测试也覆盖了 `workspace_not_found`、`busy`、`running`、`completed`、`error`、`interrupt_sent`、`restarted` 等关键状态，并通过 `result.data` 断言字段值。因此设计不能要求重写这套结构化契约，也不能随意改字段名、删字段、包裹成新的外层对象。

第四类约束来自你强调的类比参照，也就是现有工具的表现面风格。`apply_patch`、`search_text`、`read_file`、`list_directory` 之所以模型可读，不是因为它们没有结构，而是因为它们在展示给模型之前已经做过一层专门的表达设计：先给结论，再给稳定的显式层级，再给有必要的细节，避免让模型直接面对原始底层对象。这是我这次的直接设计参照。

## 3. 当前系统的真正问题

当前 `loommux` 七个工具在 `mcp_ipython_server.py` 中都是直接 `return adapter.xxx(...)`。`adapter` 返回的是 `dict[str, Any]`。按 FastMCP 默认行为，这意味着：

1. `structuredContent` 会得到这个字典对象。
2. `content` 也会得到一个由默认 serializer 生成的文本块，文本内容大体上就是 JSON 风格字符串。

这在普通 MCP 客户端里不一定是灾难，因为不少客户端还能消费 `structuredContent`。但在 `model_content_only = true` 场景下，问题马上放大，因为模型主视图只看 `content`。这时 default serializer 生成的那段 JSON 文本，就等于你的工具“实际告诉模型的话”。

这段默认文本有四个问题。

第一，结论不前置。模型每次都得先穿过一堆键值对，自己归纳“这次到底成功了、失败了、还是还在跑”。

第二，信息层级不稳定。所有字段都是平铺 JSON，模型必须靠键名自己重建层次，而不是一眼看到“结论”“kernel 状态”“输出块”“错误块”。

第三，空字段噪声大。比如 `stdout: ""`、`stderr: ""`、`error: null` 这种字段，在 JSON 里同样占据视觉重量，但对模型推理价值很低。

第四，长输出缺少语义分段。`stdout`、`stderr`、`result_text`、traceback 虽然都在结构里，但没有经过“这是标准输出”“这是标准错误”“这是表达式结果”“这是异常 traceback”的显式文本编排。模型能读懂，但读得吃力，也更容易在后续推理里丢失层次。

因此，当前问题不在 `adapter` 的状态对象本身，而在“把这个对象直接拿去做模型侧文本”的这一步。

## 4. 核心设计原则

这套 pretty text 序列化必须遵守以下原则。

### 4.1 双通道职责分离

`structuredContent` 负责机器可读、字段稳定、测试友好、程序消费；`content` 负责模型可读、层次清晰、快速扫读。两者不是竞争关系，而是分工关系。任何把二者混成一个通道的设计，都会让其中一方受损。

### 4.2 `adapter` 不承担文案责任

`adapter.py` 当前是纯状态和执行层。它负责工作区校验、kernel 生命周期、execution 状态更新、输出收集和错误表面。这个边界是对的，不能为了做漂亮文本而把一堆展示字符串塞进 `adapter`。否则你会得到两个问题：一是对象边界被污染，二是未来如果还要给别的客户端或 CLI 做别的展示面，会发现 `adapter` 里掺进了面向特定消费端的话术。

因此，展示层应该放在 server 侧，也就是 `mcp_ipython_server.py` 附近，最好抽成独立的 `presentation.py` 或 `serialization.py` 模块。

### 4.3 结论先于明细

每个工具的 `content` 文本都要让模型先看到一个结论句，而不是先看到一团字段。比如：

- `workspace` 已设置，`kernel` 已启动。
- Python execution 仍在运行。
- Python execution 已完成。
- Python execution 返回错误。
- 当前没有可读取的 execution。
- 已向当前 execution 发送中断信号。
- `kernel` 已重启。

这不是文案装饰，而是推理优化。模型拿到结果后的第一反应应该是“状态是什么”，而不是“我需要先读 12 个字段自己归纳状态”。

### 4.4 字段顺序必须固定

同类信息必须固定排序。比如在 `run_python` / `wait_python` / `read_python_output` 中，顺序始终应该是：结论、execution 标识、kernel 状态、result_text、stdout、stderr、error。顺序一旦稳定，模型跨多次工具调用就会形成固定扫读习惯，推理成本明显下降。

### 4.5 文本压缩不等于内容丢弃

这点非常关键。设计文档已经写明：适配层不截断 `stdout`、`stderr`、`result_text`。如果在 `content` 这一层为了“好看”而随意截断长输出，那么在 `model_content_only = true` 模式下，模型实际上就失去了原本应看到的完整输出，这会变成隐形削弱。故本设计不对非空输出做任意截断。可以省略空段、可以压缩无意义重复标签，但不能在没有单独规格的情况下私自裁掉实际输出内容。

### 4.6 中文主叙事，术语保留英文

你已经给出方向：长句中文叙事为主，`workspace`、`kernel`、`execution_id`、`stdout`、`stderr`、`result_text`、`KeyboardInterrupt` 这类术语保留。这样既能让文字自然，又不牺牲技术精度。

### 4.7 错误优先级高于普通状态

任何失败场景，`status` 和 `message` 必须在文本最前面就明确暴露。失败文本不能把关键信号埋在正文中后段，更不能让模型先读环境细节再知道“其实失败了”。

## 5. 模块边界设计

### 5.1 不改 `adapter.py` 的返回结构

`adapter.py` 保持现在这样：工具逻辑返回原始状态字典。原因如下。

1. 文档和测试已经围绕这个返回面建立。
2. 这是最干净的核心状态对象，适合继续作为结构化契约。
3. 未来不管是 CLI、别的 host、还是诊断测试，都还能直接复用这份原始状态。

### 5.2 在 server 层引入 presentation 模块

新增一个模块，例如：

- `src/loommux/presentation.py`
- 或 `src/loommux/serialization.py`

它的职责只有两个：

1. 把原始 `dict` 状态对象格式化成 pretty text。
2. 把 pretty text 和原始 `dict` 组装成 `ToolResult`。

不要让它碰 kernel、不碰 execution 管理、不碰工作区解析。它只做“状态对象 -> 模型文本”的变换。

### 5.3 `mcp_ipython_server.py` 改成调用 wrapper

原来是：

- `return adapter.set_workspace(path)`
- `return adapter.run_python(code, timeout_seconds)`

改成：

- `return present_set_workspace(adapter.set_workspace(path))`
- `return present_run_python(adapter.run_python(code, timeout_seconds))`

也可以统一成：

- `return present_tool_result("set_workspace", adapter.set_workspace(path))`
- `return present_tool_result("run_python", adapter.run_python(code, timeout_seconds))`

我更推荐“统一入口 + 分工具 formatter”模式，因为七个工具虽然风格一致，但字段不完全相同。

### 5.4 运行时返回 `ToolResult`，保留现有注解

为了尽量不触碰 tool schema 的元信息，我建议函数签名的返回注解先保持现在的 `-> dict[str, Any]`，只是在运行时返回 `ToolResult`。原因是：

1. 已验证 FastMCP 接受这种模式。
2. 这样不需要修改 tool output schema 的自动推导路径。
3. 现有 `list_tools` 相关测试不会受到 schema 侧额外扰动。

如果后续确定 output schema 完全不重要，再考虑把返回注解切成 `ToolResult`。但第一版没有必要为了“语义更纯”去制造元数据变化。

## 6. 统一文本骨架

我建议所有工具输出都共享一个统一的文本骨架，这样模型能快速形成稳定阅读习惯。

统一骨架如下。

第一行：结论句。

第二块：关键状态键值，采用稳定标签顺序。

第三块及以后：按语义分段展示可选内容，例如 `result_text`、`stdout`、`stderr`、`error`。

文本风格要求如下。

1. 第一行必须是一句完整结论，而不是标签列表。
2. 键值行一律单行单义，避免一行塞太多概念。
3. 非空输出段用显式段标题分隔，例如 `stdout:`、`stderr:`、`result_text:`、`error:`。
4. 如果某一块为空，就直接省略，不输出空标题。
5. 输出块正文原样保留，不再做 JSON 转义式的包裹表现。
6. 如果 `error.traceback` 存在，整段完整保留，不摘要成一句“见 traceback”。
7. 对布尔或空值的说明统一成稳定中文，而不是时而 `true/false` 时而“是/否”。建议文本里说“busy: yes/no”或“busy: true/false”保持术语一致；我更倾向于 `busy: true/false`，因为这更贴近现有状态模型。

## 7. 七个工具的具体文本设计

下面给出每个工具的最终展示面设计。

### 7.1 `set_workspace`

#### 成功

第一行：

`workspace` 已设置，`kernel` 已启动。

键值区顺序：

- `workspace: ...`
- `python: ...`
- `kernel_started: true`
- `kernel_pid: ...`
- `busy: false`
- `current_execution_id: null`
- `execution_count: 0`

如果将来切换 workspace 成功，不需要额外强调“旧 kernel 已关闭”，因为那是执行规则，不是当前结果主体。只有当用户在后续推理里需要知道时，模型可以从新的 `kernel_pid`、空 execution 状态中自然判断。

#### 失败

第一行：

`workspace` 设置失败。

第二行开始：

- `status: workspace_not_found` 或其它失败码
- `message: ...`
- `workspace: ...`
- `python: ...`

失败时不输出空的 `kernel_started`、`busy` 等字段，除非原始状态对象本身携带这些字段。对于当前实现，失败返回不带这些字段，因此文本也不凭空加。

### 7.2 `run_python`

这是最重要的工具，文本设计必须最稳。

#### `completed`

第一行：

Python execution 已完成。

第二块：

- `execution_id: ...`
- `status: completed`
- `kernel_busy: false`
- `kernel_pid: ...`
- `execution_count: ...`

之后按顺序附加非空内容：

- `result_text:`
- `stdout:`
- `stderr:`

如果 `result_text` 和 `stdout` 同时有内容，不做任何压缩取舍，顺序固定为 `result_text` 在前，`stdout` 在后。原因是 `result_text` 更接近表达式结果，是模型后续推理里更高价值的部分。

#### `running`

第一行：

Python execution 仍在运行。

第二块：

- `execution_id: ...`
- `status: running`
- `kernel_busy: true`
- `kernel_pid: ...`
- `execution_count: ...`

之后附加已产生的非空内容：

- `result_text:`
- `stdout:`
- `stderr:`

这里不能把“仍在运行”写成暧昧描述，必须明确，因为这直接影响 agent 是否继续 `wait_python`、`read_python_output` 或 `interrupt_python`。

#### `error`

第一行：

Python execution 返回错误。

第二块：

- `execution_id: ...`
- `status: error` 或 `interrupted`
- `kernel_busy: false`
- `kernel_pid: ...`
- `execution_count: ...`

然后按顺序：

- `result_text:`（如有）
- `stdout:`（如有）
- `stderr:`（如有）
- `error:`

`error:` 块中必须至少包含：

- `ename: ...`
- `evalue: ...`
- `traceback:`（完整多行）

如果状态是 `interrupted`，第一行应改为：

Python execution 已被中断。

但错误区仍可保留 `KeyboardInterrupt` 信息，因为这对模型判断执行路径非常有帮助。

#### `busy`

第一行：

当前 `kernel` 正在执行其它代码，新的 execution 未提交。

第二块：

- `status: busy`
- `current_execution_id: ...`
- `message: kernel is already executing code`

这种情况不需要虚构其它字段，因为根本没有新的 execution。

#### 其它前置失败

例如 `workspace_not_set`、`kernel_not_started`、`invalid_code`、`invalid_timeout`。

第一行：

Python execution 未开始。

第二块：

- `status: ...`
- `message: ...`

这类失败不该伪装成 execution 已经存在，更不该生成假的 `execution_id`。

### 7.3 `python_status`

这是纯状态卡。

第一行分两种：

- 如果 `workspace` 未设置：当前没有已设置的 `workspace`。
- 如果 `workspace` 已设置且 `kernel_started = false`：当前 `workspace` 已记录，但 `kernel` 未启动。
- 如果 `workspace` 已设置且 `kernel_started = true`：当前 `kernel` 可用。

键值顺序固定：

- `workspace: ...`
- `python: ...`
- `kernel_started: ...`
- `kernel_pid: ...`
- `busy: ...`
- `current_execution_id: ...`
- `execution_count: ...`
- `last_execution_id: ...`

这里不需要分出 `stdout` 或其它块，因为本工具没有输出缓存语义。

### 7.4 `read_python_output`

这是 execution 输出快照读取工具，文本上应该与 `run_python`/`wait_python` 尽量同构，让模型跨工具迁移认知。

第一行：

- `status = running`：当前 execution 输出快照已读取，execution 仍在运行。
- `status = completed`：当前 execution 输出快照已读取，execution 已完成。
- `status = error`：当前 execution 输出快照已读取，execution 返回错误。
- `status = killed`：当前 execution 输出快照已读取，execution 已被终止。
- `execution_not_found`：没有可读取的 execution。

键值顺序：

- `execution_id: ...`
- `status: ...`

之后同样按 `result_text`、`stdout`、`stderr`、`error` 顺序附加非空内容。

注意，`read_python_output` 不自带 kernel 状态字段，因此文本也不额外发明 `kernel_busy` 区块。文本必须忠实于当前工具返回对象，不要擅自借别的接口补字段，避免输出面和结构面失配。

### 7.5 `wait_python`

它与 `read_python_output` 的主要区别，是末尾多了 kernel 状态。因此文本上应该在 execution 区之后，再附一个 kernel 区。

第一行规则和 `read_python_output` 同构。

键值区：

- `execution_id: ...`
- `status: ...`
- `kernel_busy: ...`
- `kernel_pid: ...`

然后附加 `result_text`、`stdout`、`stderr`、`error`。

### 7.6 `interrupt_python`

这是动作反馈，不是状态快照。

#### `interrupt_sent`

第一行：

已向当前 execution 发送中断信号。

第二块：

- `status: interrupt_sent`
- `execution_id: ...`
- `kernel_pid: ...`

#### `idle`

第一行：

当前 `kernel` 处于 idle 状态，没有 execution 需要中断。

第二块：

- `status: idle`
- `kernel_pid: ...`

#### `kernel_not_started`

第一行：

当前没有可中断的 `kernel`。

第二块：

- `status: kernel_not_started`
- `message: ...`

### 7.7 `reset_python`

这是另一个动作反馈。

#### 成功

第一行：

`kernel` 已重启。

第二块：

- `status: restarted`
- `workspace: ...`
- `python: ...`
- `kernel_started: true`
- `kernel_pid: ...`
- `busy: false`
- `current_execution_id: null`
- `execution_count: 0`

#### 失败

第一行：

`kernel` 重启失败。

第二块：

- `status: ...`
- `message: ...`
- `workspace: ...`
- `python: ...`

## 8. 输出正文块的细化规则

### 8.1 `result_text`

`result_text` 是表达式结果或 display 文本。它在很多场景比 `stdout` 更接近“最终答案”，所以应优先展示。展示格式为：

`result_text:`
随后直接跟原文，多行原样保留。

### 8.2 `stdout`

展示格式为：

`stdout:`
随后原样保留所有文本，不额外包 JSON 字符串，不自行截断。

### 8.3 `stderr`

展示格式为：

`stderr:`
随后原样保留所有文本。

### 8.4 `error`

`error` 不是一句话，而是一个对象。文本化时不能只保留 `ename`，否则 `evalue` 和 traceback 细节会丢失。建议格式：

`error:`
`ename: ...`
`evalue: ...`
`traceback:`
后面完整展开 traceback 列表内容。

### 8.5 空内容省略规则

如果某段为空，整段省略。例如：

- `stdout == ""`，则不输出 `stdout:` 标题。
- `error is null`，则不输出 `error:` 段。

这样可以明显减少空值噪声。

### 8.6 不增加新的截断参数

这次设计绝不引入 `max_output_chars`、`truncate`、`preview_lines` 等参数，因为设计文档已明确禁止这种接口扩展。展示层也不引入隐藏截断，只做排版和分段，不做内容删减。

## 9. 具体实现方式

### 9.1 新增 formatter 函数族

建议新增一组纯函数，例如：

- `format_set_workspace_content(status: dict[str, Any]) -> str`
- `format_run_python_content(status: dict[str, Any]) -> str`
- `format_python_status_content(status: dict[str, Any]) -> str`
- `format_read_python_output_content(status: dict[str, Any]) -> str`
- `format_wait_python_content(status: dict[str, Any]) -> str`
- `format_interrupt_python_content(status: dict[str, Any]) -> str`
- `format_reset_python_content(status: dict[str, Any]) -> str`

这些函数只依赖传入状态对象，不碰全局状态，不做 I/O。这样测试非常简单，也便于以后迭代展示面。

### 9.2 新增统一包装器

新增一个小包装器：

`present_status(tool_name: str, status: dict[str, Any]) -> ToolResult`

它内部做三件事：

1. 根据 `tool_name` 选择对应 formatter。
2. 生成 pretty text。
3. 返回 `ToolResult(content=pretty_text, structured_content=status)`。

### 9.3 `mcp_ipython_server.py` 只接包装器

七个工具都改为：

1. 先调 `adapter` 拿原始状态字典。
2. 再调 `present_status(...)`。
3. 返回 `ToolResult`。

这会使 server 文件从“直接透传 adapter 状态”变成“adapter 状态 + 表现层包装”，但不会污染 `adapter` 本身。

### 9.4 保持输入 schema 不变

工具参数和输入 schema 不变；七个工具名不变；不新增 `start_python`；不新增输出截断参数。这一点是硬约束。

## 10. 测试设计

当前测试主要盯 `result.data`。这非常好，因为它能让我们在新增 pretty text 的同时保持结构契约回归验证。新增测试应分三层。

### 10.1 结构契约回归层

现有所有黑盒测试保留，继续确保：

- `result.data` 的字段和值不变。
- 七个工具行为不变。
- 文档承诺的状态码和状态流不变。

### 10.2 `content` 展示面测试层

新增一批测试，直接验证 `result.content` 中的文本。

应该至少覆盖：

1. `set_workspace` 成功文本。
2. `set_workspace` 失败文本。
3. `run_python` 的 `completed` 文本。
4. `run_python` 的 `running` 文本。
5. `run_python` 的 `error` 文本。
6. `busy` 文本。
7. `python_status` 的无 workspace 文本。
8. `read_python_output` 的 `execution_not_found` 文本。
9. `interrupt_python` 的 `interrupt_sent` 文本。
10. `reset_python` 的 `restarted` 文本。

测试方法不需要全文 snapshot 到每个字符，但必须验证关键段存在和顺序稳定。例如检查：

- 第一行结论句是否正确。
- `status:` 行是否出现。
- `workspace:`、`python:`、`execution_id:` 等关键行是否出现。
- `result_text:`、`stdout:`、`stderr:`、`error:` 是否按预期出现或省略。

### 10.3 formatter 纯函数单元测试层

如果 formatter 抽到独立模块，再补纯函数测试会非常省事。给定一个固定状态字典，直接断言生成文本。这样不需要每次都走 MCP client。

这层测试的价值在于：它能快速锁定“文本设计回归”，不必每次都通过完整 server 调用链排查。

## 11. 为什么不选其它方案

### 11.1 不选“只回纯文本”

因为这会直接放弃结构化契约，现有 `result.data` 路径和黑盒测试都会失去意义。它虽然最省事，但不专业。

### 11.2 不选“在 adapter 里直接拼文本”

因为那会污染核心层，让状态对象和展示话术耦合。以后只要展示面想改，核心层也得动，边界会迅速变脏。

### 11.3 不选“让 `content` 继续吃默认 JSON，再指望模型自己理解”

因为这正是当前问题。`model_content_only = true` 的意义就是：模型优先看到 `content`。如果 `content` 没有被设计，那这个开关本身就被浪费了。

### 11.4 不选“私自截断 stdout/stderr/result_text 来提升可读性”

因为这与当前文档精神冲突，而且会让模型在最需要输出全文时反而拿不到完整信息。没有单独规格之前，不做展示层截断。

## 12. 风格细则

为了让这套文本真的像工具而不是像散文，风格上再约束几条。

1. 第一行永远是结论句，不写“以下是本次执行结果”。
2. 标签统一使用英文术语，叙事统一中文。
3. 行文不使用夸张、安抚、建议式语气，不出现“建议你接下来”“你可以考虑”。工具只报告结果，不当助手口播。
4. 错误文本不用感叹句，不做人类化情绪表达。
5. 所有文本都应在没有 JSON 语法高亮的情况下仍然清晰可扫读。
6. 标题和段落不要过多，避免把工具结果写成说明书。它必须短，但结构清晰。
7. 如果某个字段不存在，就不造。文本必须忠实于状态对象。

## 13. 实施顺序

真正开工时，我会按下面顺序做，而不是同时乱改。

1. 新增 `presentation.py`。
2. 先写共用小工具，例如空段过滤、键值块拼装、错误块拼装、输出块拼装。
3. 再写七个 formatter。
4. 修改 `mcp_ipython_server.py` 接入 `ToolResult` 包装。
5. 保持 `adapter.py` 不动，除非实现中发现非常小的协助性需求。
6. 新增 `content` 文本测试。
7. 运行现有黑盒测试，确保 `result.data` 完全不变。
8. 运行 `ruff`、`basedpyright`、`pytest` 全量验证。
9. 如果需要，再用一个最小 client 探针打印 `result.content`，人工目测可读性。

## 14. 预期效果

做完之后，模型看到的工具结果会发生质变。

以前模型看到的，是类似：

`{"ok": true, "status": "running", "execution_id": "exec-000002", "stdout": "0\n", "stderr": "", "result_text": "", "error": null, ...}`

以后模型看到的，会是类似：

`Python execution 仍在运行.`
`execution_id: exec-000002`
`status: running`
`kernel_busy: true`
`kernel_pid: 12345`
`execution_count: 3`
`
stdout:`
`0`

结构化消费侧仍然保持原始字典，不会被这层文本美化破坏。也就是说，模型阅读体验改善，而程序契约不退化。

## 15. 最终拍板

这份设计的核心判断只有一句：

`loommux` 不应该继续把 `dict` 的默认字符串化结果当作 `content`，而应该像成熟工具一样，把 `content` 明确设计成面向模型的 pretty text；同时保留当前状态字典作为 `structuredContent`，形成稳定双通道输出。

这不是可选优化，而是在 `model_content_only = true` 条件下的必要修正。

如果按这份设计开工，实施层面不会去重新讨论方向，只会做三件事：

1. 新增 presentation/serialization 层。
2. 让七个工具返回 `ToolResult(content=pretty_text, structured_content=raw_status_dict)`。
3. 补 `content` 文本测试并回归现有结构化测试。

这就是最终设计。
